# 2026-03-26 工作笔记：改签工具开发

## 一、方案讨论

### 核心结论

- **查询**：复用已有的 `MySQLQuery` tool（SELECT，只读账号）
- **写操作**：新增 `RebookExecuteTool`，使用专用写库 MySQL 账号，代码层强校验
- **流程编排**：新建 `skills/rebook-order/SKILL.md`

### 改签特殊性

比退款更复杂，原因：
1. 改签字段组合多（时间 / 出发站 / 到达站 / 座位，可任意组合）
2. 双程订单需判断改哪一程（智能判断：去程已验票/已完成 → 默认改返程）
3. 改站点时地址数据来源——从现有订单中找相同站点的记录复制省市区/坐标（不从 lulu_station 表取）
4. 返程改签需额外同步更新 `lulu_order_attachment`（旧表，未下线）

### 写库安全方案

不开全量写权限，专用账号 + 代码三重限制：

| 层次 | 限制 |
|---|---|
| MySQL 用户权限 | 只有 UPDATE，无 INSERT/DELETE/DROP |
| 表白名单 | 只允许 4 张表 |
| 字段白名单 | 只允许改签相关字段，财务/状态/id 字段全部拒绝 |
| 强制 WHERE | order/itinerary/attachment 必须含 `order_no`；ticket 必须含 `order_id` |
| 事务 | 多表一次提交，任意一条失败全部回滚 |

## 三、RebookExecuteTool 设计要点

接受 `updates` 数组，支持一次调用更新多张表，单事务提交：

```json
{
  "updates": [
    { "table": "lulu_order",           "set": {"departure_time": "2026-03-27 09:00:00"}, "where": {"order_no": "GZ2026032790059"} },
    { "table": "lulu_order_itinerary", "set": {"departure_time": "2026-03-27 09:00:00"}, "where": {"order_no": "GZ2026032790059", "itinerary": 1} },
    { "table": "lulu_ticket",          "set": {"seat": "9"},                              "where": {"order_id": 12345, "itinerary": 1} }
  ]
}
```

各表允许更新的字段：

| 表 | 允许字段 | 强制 WHERE |
|---|---|---|
| lulu_order | departure_time、departure/arrive 省市区地址经纬度站点ID、schedule_Id | order_no |
| lulu_order_itinerary | 同上 | order_no |
| lulu_order_attachment | 同上 | order_no |
| lulu_ticket | seat（仅此一个） | order_id |

使用独立配置 `mysql_write_user` / `mysql_write_password`，复用 `mysql_host`/`mysql_port`/`mysql_database`。

## 四、SKILL.md 改签流程设计要点

### 判断改签哪一程（双程逻辑）

| 去程状态 | 处理 |
|---|---|
| state=260（已验票）或 300（已完成） | 默认改返程，告知用户确认 |
| 其他 | 询问用户改哪一程 |

### 哪一程决定更新哪些表

| 改签程次 | 更新表 |
|---|---|
| 单程 / 去程 | lulu_order + lulu_order_itinerary(itinerary=1) + lulu_ticket(itinerary=1) |
| 返程 | lulu_order_itinerary(itinerary=2) + lulu_order_attachment + lulu_ticket(itinerary=2) |

### 站点地址参考数据查询

改出发站：在 lulu_order（去程）或 lulu_order_itinerary itinerary=2（返程）里 `LIKE '%站点名%'` 找一条，复制 province/city/district/addr/position/schedule_station_id。

改到达站：同理，查 arrive_* 字段。

若找不到历史记录，让用户手动提供完整地址信息。

### 其他要点

- `schedule_Id` 不强制更新（改签不一定换班次）
- `lulu_ticket` 只更新 `seat`（ticket 表无 departure_time 和地址字段）
- 执行前必须展示变更前后对比并让用户确认
